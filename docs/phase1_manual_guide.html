0 · Concepts first — RAG, embeddings & vector DBs

Skip this section if you already build RAG systems. If you don't, five minutes here makes every command below make sense.

What are we building, in one picture?

A user asks a plain-English question about a company's financial filings and gets an answer where every sentence carries a citation to the real document — or the system refuses if it lacks the evidence. No invented facts. The technique that makes this possible is RAG — Retrieval-Augmented Generation:

RAG = an open-book exam

Instead of asking a language model to answer from memory (a closed-book exam, where it will confidently invent a number it doesn't remember), we first retrieve the most relevant paragraphs from the real documents, paste them into the prompt, and say: "answer using only this, and cite each claim." Everything in Phase 1 exists to build that "open book" and to find the right page fast.

Jargon → concepts you may already know (this project uses a CV analogy)

If you know this (computer vision / ML)…then this RAG term is the same idea
A CNN turns an image into a feature vectorAn embedding model turns a paragraph into a vector
Similar images → similar feature vectorsSimilar meanings → similar text vectors
Cosine similarity to compare feature vectorsThe exact same metric, to compare meaning
FAISS / a nearest-neighbour indexA vector database (here: Postgres + pgvector)
Tiling a big image into patches before processingChunking a 300-page document into passages

What is an embedding?

A list of numbers (here, 3072 of them) that captures the meaning of a piece of text. The key property: texts with similar meaning get similar vectors even if they share no words. "loan loss provision" and "money set aside for bad loans" land close together; "rooftop garden" lands far away. Exactly like a CNN mapping similar images to nearby feature vectors — same idea, applied to text.

How is a vector database different from a normal database?

Traditional database (what most people know)Vector database (what RAG needs)
Storesrows of numbers, dates, stringsrows plus high-dimensional vectors
You ask it"rows WHERE ticker = 'JPM'" (exact match)"the 8 rows whose vector is nearest to this one"
Question typeexact / filteringsimilarity / nearest-neighbour, ranked
CV analogya plain SQL tablea FAISS index

The good news: we don't need an exotic new system. We use PostgreSQL (a rock-solid traditional database) plus an extension called pgvector that adds a vector column type and nearest-neighbour search. So one database does both jobs — exact filters (WHERE ticker = 'JPM') and similarity search (ORDER BY embedding <=> query_vector, where <=> is pgvector's "distance between vectors" operator).

Why chunking?

Two reasons: (1) embedding models have an input-size limit — you can't turn a 112,000-character section into one vector; and (2) retrieval works best on focused passages — when someone asks about ransomware you want the paragraph about ransomware, not a 40-page section. You don't feed a 10,000×10,000 image to a network as one tensor — you tile it into patches. Chunking is tiling, for text.

One clarification that confuses everyone: HTML vs JSON

The SEC gives us the filing as HTML (a .htm web page). The .jsonl files you'll see later (JPM_10-K_2025-12-31.jsonl) are not from the SEC — we create them, after cleaning and chunking. They are our processed output, not the raw input. (Separately, the SEC's index lookups — "which filings exist" — also return JSON, but the filing content is HTML.) In short: SEC gives HTML → our code produces JSON.

That's the whole conceptual toolkit. The 8 steps below are just this picture, made real:

SEC EDGAR ──▶ 1.download HTML ─▶ 2.clean text ─▶ 3.split sections ─▶ 4.chunk (+tags)

   answer ◀─ 8.LLM writes cited answer ◀─ 7.search DB ◀─ 6.embed ◀─ 5.store rows

3 · Start the database (Docker + Postgres + pgvector)

The vector store is ordinary Postgres 16 with the pgvector extension, declared in infra/docker-compose.yml. Three things worth understanding before you run it:

  • The image pgvector/pgvector:pg16 is official Postgres with the extension pre-installed — nothing to compile.
  • The schema applies itself. The compose file mounts infra/schema.sql into /docker-entrypoint-initdb.d/; Postgres executes anything in that folder on first boot with an empty data volume. That's why you never run a migration by hand.
  • Data survives restarts in the named volume finsight_pgdata.
docker compose -f infra/docker-compose.yml up -d

First run downloads the image (~150 MB), then starts a container named finsight-db listening on localhost:5432 (user/password/db all finsight).

docker exec finsight-db psql -U finsight -d finsight -c "\d chunks" should print the chunks table definition, including a embedding | vector(3072) column and a generated tsv | tsvector column.

What the schema means

Column(s)Purpose
ticker, form, period, item, seqMetadata for SQL filtering; together they form a UNIQUE constraint so re-loading never duplicates rows (idempotent loads)
citationPre-built source label, e.g. [JPM 10-K 2025, Item 1A] — travels with the chunk into the final answer
textThe chunk content, metadata header included
embedding vector(3072)The semantic vector. No ANN index on purpose: pgvector indexes cap at 2000 dims; at this scale a sequential scan is milliseconds
tsv tsvector + GIN indexPre-built full-text column for Phase 2's BM25 hybrid search — costs nothing now

5 · Run the data pipeline: extraction → preprocessing → chunking

This step needs no API key and costs nothing — it only talks to the free SEC EDGAR API.

.venv\Scripts\python -X utf8 -m ingestion.pipeline JPM BAC --forms 10-K
Expected output (report dates may differ): 
JPM: 1 filings
  10-K 2025-12-31: 12 sections -> 475 chunks
BAC: 1 filings
  10-K 2025-12-31: 16 sections -> 256 chunks

What just happened, stage by stage:

5a · Extraction (ingestion/edgar_client.py)

  1. Downloads SEC's company_tickers.json to map ticker → CIK (the SEC's company ID: JPM → 19617).
  2. Fetches data.sec.gov/submissions/CIK0000019617.json — the company's filing index — and filters to the requested forms and date range.
  3. Downloads each filing's primary document into data/raw/<TICKER>/ (JPM's FY2025 10-K is ~12.9 MB of inline-XBRL HTML — this is the raw input; see the HTML-vs-JSON note in Section 0) with a small JSON sidecar of filing metadata.

Compliance details baked in: a real User-Agent header (SEC requirement) and a throttle keeping requests under ~8/second (SEC caps at 10/s). Downloads are cached — rerunning is instant.

5b · Preprocessing (ingestion/parser.py)

  1. HTML → text: strips <script>/<style>, then flattens every <table> into pipe-separated rows — Provision for credit losses | 9,320 | 6,899 — so a number keeps its row label as context. Whitespace is collapsed but line breaks kept (heading detection needs them).
  2. Text → named sections: a regex finds Item N. headings (Item 1A = Risk Factors, Item 7 = MD&A…). The trap: every item also appears in the table of contents. Fix: a match with <200 chars of body before the next match is a TOC entry — dropped; if an item still matches twice, the longest occurrence wins.
Known quirk: JPMorgan "incorporates by reference" — its MD&A and financial statements physically live inside a ~1M-char blob after Item 15. The content is indexed, just labeled Item 15. Documented in the README; fixing the labeling is Phase 2 fodder.

5c · Chunking (ingestion/chunker.py)

  1. Each section splits into ~4,000-char chunks on paragraph boundaries (never mid-sentence), with a one-paragraph overlap between consecutive chunks so boundary sentences are never orphaned.
  2. Every chunk is prefixed with a metadata header the embedding model will see: 
    [JPM | 10-K FY2025 | Item 1A: Risk Factors]
  3. Every chunk carries a ready-made citation string: [JPM 10-K 2025, Item 1A].

Why not fixed-size chunking: it slices through tables and section boundaries, destroying the context that makes a passage retrievable. Section-aware chunking with metadata headers is the single highest-leverage design decision in Phase 1.

6 · Inspect what the pipeline produced

Output lands as JSON-lines files, one per filing, in data/parsed/:

dir data\parsed
:: JPM_10-K_2025-12-31.jsonl   BAC_10-K_2025-12-31.jsonl

Look at one chunk (each line is one JSON object):

.venv\Scripts\python -X utf8 -c "import json; c=json.loads(open('data/parsed/JPM_10-K_2025-12-31.jsonl',encoding='utf-8').readline()); print(json.dumps({k:(v[:200]+'...' if k=='text' else v) for k,v in c.items()}, indent=2))"
You should see fields ticker, form, period, fiscal_label, item, section_name, seq, text, citation — and text starting with the [JPM | 10-K FY2025 | ...] metadata header.

7 · Embed the chunks into the database

.venv\Scripts\python -X utf8 -m ingestion.embedder

For each .jsonl file this batches chunks (64 at a time) to the embedding model and inserts rows with ON CONFLICT DO NOTHING — so rerunning after a crash resumes safely.

On free-trial rate limits this takes 10+ minutes for ~700 chunks and prints nothing while working — it is not stuck; the SDK is silently backing off on 429 (rate-limit) responses. Watch progress from another terminal: 
docker exec finsight-db psql -U finsight -d finsight -c "SELECT ticker, count(*) FROM chunks GROUP BY ticker;"
When finished, the count query above shows JPM 475, BAC 256 (or similar for your filing vintages), and the embedder printed one "N new chunks embedded" line per file.

8 · Ask questions — the cited RAG CLI

.venv\Scripts\python -X utf8 rag_cli.py "What cybersecurity risks did JPMorgan flag?" --tickers JPM

Flow: your question is embedded with the same model → cosine-distance search in pgvector (ORDER BY embedding <=> query) filtered to JPM → the top 8 chunks are pasted into a prompt whose rules are: use only this evidence, cite every claim, refuse if the evidence doesn't answer.

Expected: a multi-bullet answer where every bullet ends in a citation like [JPM 10-K 2025, Item 1A], followed by the retrieved sources with similarity scores (~0.65–0.75).

Now the trust test — ask something the index cannot answer:

.venv\Scripts\python -X utf8 rag_cli.py "What did Tesla say about vehicle recall risks?" --tickers JPM BAC
Expected: "Insufficient evidence in the indexed filings." plus an explanation of what's missing — and retrieval scores collapsing to ~0.25. The system refuses rather than hallucinates. This behavior is the entire point of the architecture.

9 · Daily restart routine

After a reboot, only one thing needs manual attention:

  1. Start Docker Desktop, wait for it to finish loading.
  2. docker start finsight-db (or the compose up -d command again).

Everything else persists: the venv, .env, downloaded filings in data/, and all database rows in the finsight_pgdata volume. Azure deployments are pay-per-token — idle costs nothing.
