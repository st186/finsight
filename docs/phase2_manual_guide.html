0 · Concepts first

The Phase 2 mindset shift

Phase 1 ended with demos that looked right. Phase 2's question is: "how good is retrieval, numerically, and does each change make it better or worse?" The ML analogy: you never ship a classifier off five demo images — you build a labelled test set and measure. Here the test set is called a golden dataset and the headline metric is hit rate: the fraction of questions where the known-correct evidence appears in the top-k retrieved chunks.

The four retrieval modes you'll compare

ModeHow it worksWins atLoses at
vector (Phase 1 baseline)embed the question, nearest-neighbour searchparaphrases, "search by meaning"exact identifiers (product names, legal phrases)
keywordPostgres full-text search (BM25-family scoring) on the tsv columnexact tokens: "iPhone 17 Pro", "Item 1A"paraphrases with zero word overlap
hybridrun both, merge rankings with Reciprocal Rank Fusionunion of both strengths—
hybrid+rerankhybrid top-24 → cross-encoder re-scores → top-8precision: the right chunk into the top slotslatency (CPU model per query)

RRF in one line

Two ranked lists have incomparable scores (cosine vs ts_rank), so ignore scores and use ranks: each document earns 1/(60+rank) from every list it appears in; sum and re-sort. A document high in both lists collects points twice and floats to the top.

Bi-encoder vs cross-encoder

The embedding model is a bi-encoder: query and document encoded separately (fast — vectors precomputed, but it never reads them together). A cross-encoder reads query+candidate as one input and outputs a relevance score (accurate but slow) — so it's used as a second stage on ~24 candidates only. Identical to detection pipelines: cheap proposals → expensive classifier head. Ours: BAAI/bge-reranker-base, plain CPU.

LLM-as-judge (answer-level metrics)

Retrieval metrics can't catch an LLM misusing good evidence. So a judge model scores each generated answer for faithfulness (claims supported by evidence?) and relevancy (question addressed?), and the 10 deliberately unanswerable questions must produce the literal "insufficient evidence" refusal — that's refusal correctness.

2 · Ingest the full 10-company corpus

.venv\Scripts\python -X utf8 -m ingestion.pipeline C GS MS WFC AAPL MSFT GOOGL NVDA --forms 10-K --since 2024-06-01

(JPM and BAC are already parsed from Phase 1.) Free, no API key.

One line per filing, e.g. GS: 10-K 2025-12-31: 19 sections -> 419 chunks. Expect ~13 filings, ~2,800 chunks total across data/parsed/.
Three real data-quality findings you WILL see (left in on purpose — this is what filings are actually like):
  • Citi & Morgan Stanley parse as 1 giant section ("Item FULL") — they format their Item headings inside HTML tables, which our regex misses after table-flattening. Content is still indexed and searchable; section filtering is degraded for those two.
  • Wells Fargo yields only ~33 chunks — its primary document is a thin wrapper; the real report lives in a separate exhibit file (incorporation by reference, in a different file than JPM's variant). Fetching exhibits is a backlog item.
  • Banks show fewer historical 10-Ks — EDGAR's submissions API returns only the ~1000 most recent filings and heavy filers flood that window.
The eval in step 5 measures the impact of these instead of hiding them.

3 · Embed everything (and survive rate limits)

.venv\Scripts\python -X utf8 -m ingestion.embedder

~2,800 chunks through text-embedding-3-large. On a trial-tier Azure deployment this takes a long time and will hit hard 429 rate limits. The embedder now: (1) waits out the quota window (65s sleeps, up to 10 times) instead of dying, and (2) commits per batch, so an interrupted run resumes exactly where it stopped — just re-run the same command.

Seeing 429 rate limit — sleeping 65s lines is normal, not a failure. Progress-watch from another terminal: 
docker exec finsight-db psql -U finsight -d finsight -c "SELECT ticker, count(*) FROM chunks GROUP BY ticker ORDER BY ticker;"
Final counts show all 10 tickers. Ballpark: JPM 475, C 474, GS 419, MS 358, BAC 256, GOOGL ~236, MSFT ~243, NVDA ~193, AAPL ~131, WFC ~69 — roughly 2,850 rows.

4 · Understand the golden dataset

evals/golden_dataset.jsonl — 44 questions, one JSON per line:

{"id": "sf-03", "type": "single_fact",
 "question": "What emerging technology does JPMorgan say may require investments in new encryption protections?",
 "tickers": ["JPM"],
 "expected_phrases": ["quantum-resistant encryption"],
 "answerable": true, ...}
TypeTestsCount
single_factfind one fact in one filing18
sectionItem-level filtering (e.g. only 1A)6
comparisonevidence from two companies in one top-k5
temporalthe right year's filing (iPhone 16 vs 17; Blackwell vs Rubin)6
unanswerablemust refuse (Tesla, forecasts, out-of-range years)10

The grounding discipline: every expected_phrases entry was copied verbatim from the corpus and verified to exist before the question was written. The metric is therefore deterministic — no human judgment at eval time. If you extend the dataset, follow the same rule: grep the phrase in data/parsed/ first, then write the question.

5 · Run the retrieval evals (the Phase 2 deliverable)

.venv\Scripts\python -X utf8 -m evals.run_evals --modes vector keyword hybrid hybrid+rerank

First run of the hybrid+rerank mode downloads the cross-encoder (~1 GB, one-time) into your HuggingFace cache. The rerank mode is the slow one — a CPU model scoring 24 candidates for each of 34 questions.

A table like this (your exact numbers will vary): 
mode            hit rate   citation acc (temporal)   misses
vector          ~7x%       ~8x%                      sf-.., cmp-..
keyword         ~5x%       ~6x%                      ...
hybrid          >= vector  >= vector                 fewer
hybrid+rerank   highest    highest                   fewest
The expected shape: hybrid ≥ both single modes; rerank converts near-misses into hits. If a "better" mode scores lower, that's a finding — investigate, don't average it away.

6 · Run the answer-level evals

.venv\Scripts\python -X utf8 -m evals.run_evals --modes hybrid+rerank --answers

Costs real LLM calls: one generated answer per question plus one judge call per answerable question (~70 calls on gpt-5-mini — cents, not dollars, but not free like step 5).

Output ends with a line like: 
hybrid+rerank answers: faithfulness=0.9x relevancy=0.9x refusals=10/10 (100% correct)
refusals=10/10 is the number to care about most — it means every trap question produced "insufficient evidence" instead of a hallucination.

7 · Reading the results — what to do with misses

  1. Look at the miss list, not just the percentage. Each miss has a question ID. Re-run that one question by hand and look at what WAS retrieved: 
    .venv\Scripts\python -X utf8 -c "from retrieval.search import retrieve; [print(f'{h.score:.3f} {h.citation} {h.text[:90]}') for h in retrieve('YOUR QUESTION', k=8, mode='hybrid+rerank', tickers=['JPM'])]"
  2. Classify the failure: (a) corpus problem — the evidence is in a mislabeled Citi/MS blob or missing WFC exhibit; (b) retrieval problem — right chunk exists but ranked below top-k; (c) golden-set problem — the expected phrase pins one wording but the fact exists in multiple places (e.g. total assets in Item 1 prose AND Item 15 balance sheet).
  3. Fix the category, not the symptom — parser fixes for (a), rewriting/reranking tuning for (b), a second accepted phrase for (c). Then re-run the eval: the number must move.
